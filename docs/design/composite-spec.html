官网与文档组合规范

这里承接从基础组件规范中拆出的 Site*、Docs* 和跨组件页面模式。它们复用同一组 Refined Sage token，但属于页面组合层，不再挤占基础组件导航。

Site*Docs*页面模式复用基础组件
基础组件层由基础规范维护，提供 Button、Input、Table、Toast 等原子能力。
组合页面层本页维护官网、文档和跨组件流程，控制页面节奏与真实场景。
宿主工作台层Workbench*、SettingsHost 等仍由产品宿主负责，不在本页重定义。

官网搭建规则

官网不是单独的视觉稿，而是基于 Tabora 基础组件向上组合的营销站点层。站点组件使用 Site* 命名，消费同一组 token，并在内部复用 Button、Input、Kbd、Badge、 Chip、Field、Table 等基础件。

Site* 命名复用基础件响应式网页官网专用组合
基础组件层 Button、Input、Chip、Badge、Kbd、Field、Toast 等可进入插件内容区和官网。
官网组合层 SiteSection、SiteTopnav、SiteHero、SiteProductPreview、SiteProof、SiteDownloadCard、DocsShell、DocsCodeBlock、DocsComponentExample、SiteCTA、SiteWaitlist、SiteFooter。
宿主工作台层 WorkbenchRail、WorkbenchGrid、SettingsHost、Modal 等由 shell/layout host 提供，官网只预览。

组件分层

  • @tabora/ui：Button、Input、Select、Badge、Kbd、Toast 等基础组件。
  • Site*：官网页面结构组件，只负责营销站点的节奏、导航、首屏、产品预览、证明和 CTA。
  • Workbench*：真实工作台宿主组件，不放进官网组件层，也不由官网直接实现业务状态。

Do

官网可以使用更大的标题和更强的产品预览，但仍坚持边框、暖白底、低频绿色强调和紧凑信息密度。

Don't

不要在官网里另起紫色渐变、暖橙背景、插画人像或和工作台无关的装饰卡片。

SiteSection 官网区块节奏

官网由独立区块组合，而不是把所有内容塞进一个长画布。每个区块必须有清晰职责和可评论的 data-od-id。

section rhythmdata-od-idsingle responsibility

HERO

先解释产品，再露出工作台。

PROOF

用协议边界证明，而不是堆数字。

相邻区块只用 1px 分隔线和留白建立节奏，不包多层卡片。

默认官网顺序

  • SiteTopnav → SiteHero + SiteProductPreview → SiteFeatureGrid。
  • SiteProof 可用矩阵或日志列表表达架构、官方插件和 MVP 范围。
  • SiteCTA + SiteWaitlist 负责收束，不新增第三个主要行动。

SiteTopnav 官网顶部导航

承载品牌、页面锚点和 1 个主操作。用于官网，不用于工作台应用内 rail 或 settings host。

品牌入口锚点导航最多 1 个 primary

Anatomy

  • .site-logo 使用 LogoMark + 产品名，不把导航标题当作 H1。
  • .site-navlinks 只放页面锚点，不放产品内部路由或设计稿切换器。
  • .site-actions 复用 Button，主按钮用于等待名单、下载或联系销售。

SiteProductPreview 产品预览框

以浏览器外壳包裹真实工作台缩略图。它是官网里的产品证据，不是独立应用组件。

组成 类名 规则
浏览器栏 .site-browser-bar 交通灯、地址、快捷键，使用低对比表面色。
工作台预览 .site-workbench 只展示核心结构，不暴露设计状态、viewport 标签或 mock 控制。
预览卡片 .site-widget 复用 8px 卡片半径和 1px 边框，内部用真实模块名。

响应式

  • 桌面端预览可放在 hero 右侧；920px 以下进入单列并保持完整宽度。
  • 360px 到 430px 下不保留 rail，转为内容区预览，避免横向滚动。
  • 预览内部元素用稳定高度，不因 hover、focus 或 toast 改变布局尺寸。

SiteFeatureGrid 价值点栅格

用于解释协议能力，每个特性必须对应 Tabora 的真实产品边界或 MVP 功能。

WHAT CHANGES

不是功能堆叠，而是可装配的工作台协议。

价值点写产品事实，不写泛化形容词。每组最多 3 项，避免把页面变成卡片墙。

布局本身也是插件

Dashboard 和 Stream 是官方布局范式，布局决定区域和入口，不接管 widget 数据。

入口在任何布局下可达

搜索、添加卡片、插件管理和设置必须通过 rail、工具条或命令面板保持可达。

失败被限制在局部

Widget、搜索源或布局插件失败时都有独立回退，不让用户面对整页白屏。

用于把官网从产品解释收束到一个明确动作。CTA 只承载一个主要目标，等待名单表单是它的可选子组件。

SiteWaitlist 转化表单

用于等待名单、产品评审或下载通知。表单必须有真实 label 或 aria-label，并复用基础输入和 Toast 反馈。

SiteToast 官网反馈

官网中的主题切换、布局切换、等待名单提交都使用 Toast 反馈。Toast 是浮层反馈，不占据正常文档流。

已切换到 Stream，实例数据保持不变。

状态映射

  • 成功：使用 --cp-success，用于等待名单提交、保存和布局切换。
  • 警告：使用 --cp-warning，用于非阻塞提示，例如网络不稳定。
  • 错误：使用 --cp-danger，用于表单格式错误或动作失败；字段错误优先用 InlineError。

SiteFooter 官网页脚

官网页脚只承载版权、产品定位和少量辅助链接。不要把页脚做成二级导航中心或设计系统说明区。

© 2026 Tabora Plugin-first personal workbench · Refined Sage system

Rules

  • 页脚和页面主体之间只用 1px 分隔线，不再包一层漂浮卡片。
  • 移动端允许左右两段换行；不要缩小到不可读字号。
  • 如果增加链接，优先使用基础 Link；最多一组 3-4 个辅助链接。
