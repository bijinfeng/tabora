## 1. 产品概述

Tabora 需要提供官方账号云同步能力，让用户在多台设备之间保持工作区结构、卡片布局、插件配置和插件声明为可同步的业务数据一致。

同步能力属于 Tabora core，而不是某个 Dashboard 卡片或普通插件。Core 负责账号、设备、同步队列、冲突检测、加密存储、快照恢复和失败兜底；设置中心负责把这些能力以可理解的方式展示给用户。

第一版同步只进入设置中心，新增一级分类「数据同步」。工作台首页不放同步卡片，只在需要用户注意时通过轻量提示、徽标或冲突入口提醒。

## 2. 背景与问题

Tabora 的本地工作台已经有多分组、插件实例、卡片布局、插件配置和插件私有数据。用户在一台设备上整理好的工作台，通常会希望在另一台设备继续使用：

- 同一账号下的工作区分组和布局保持一致。
- 插件安装状态、卡片实例和配置不需要重复搭建。
- 待办、重点、快捷入口等插件数据可以跨设备延续。
- 本机路径、密钥、缓存和临时文件不能被错误同步。

如果每个插件自行做云同步，会造成协议碎片化、隐私边界不一致、冲突处理不可控、账号体系重复建设。同步必须由 core 统一实现，插件只声明哪些数据集合可以被同步。

## 3. 已确认决策

| 主题     | V1 决策                                                               |
| -------- | --------------------------------------------------------------------- |
| 同步形态 | 官方账号云同步                                                        |
| 账号体系 | 官方账号支持注册、登录、退出登录和会话恢复；未登录时保持本地模式      |
| 产品入口 | 设置中心一级分类「数据同步」                                          |
| 首页暴露 | 不做 Dashboard 卡片，只保留轻量提醒                                   |
| 加密策略 | 服务端加密存储 + 敏感数据排除                                         |
| E2EE     | 完整端到端加密不进入 V1                                               |
| 同步范围 | 工作区结构、插件配置、插件显式声明的可同步业务数据                    |
| 敏感排除 | API Key、本机路径、缓存、临时图片、OCR 原图、导出结果、设备特有偏好   |
| 冲突策略 | 自动记录级合并，无法合并进入冲突收件箱                                |
| 设备策略 | 登录官方账号后自动注册当前设备，设置页可移除其他设备                  |
| 触发策略 | 后台自动同步为主，保留「立即同步」                                    |
| 后端平台 | Supabase（Auth + Postgres + Edge Functions）                          |
| 同步模型 | state-based 当前态：云端每行一条记录 + `updatedAt` + tombstone 软删除 |
| 访问形态 | 客户端只经 Supabase Edge Function 同步网关读写，不直连数据表          |
| 加密语义 | 平台级加密（传输 TLS + 存储 at-rest）+ 敏感字段永不上传，非 E2EE      |

### 3.1 核心技术决策（V0.2 锁定）

本节锁定影响 Supabase 表结构与同步 API 形态的三个技术决策。后续技术方案与实现以此为准。

**决策一：云端事实源采用 state-based 当前态模型。**

- 云端为每个可同步实体保存「当前状态」单行记录，携带稳定 `recordKey`、`updatedAt` 和软删除标记（tombstone：保留主键 + `deletedAt`，不物理删除）。
- 合并语义即 state-based：`last-write-wins` 比较 `updatedAt`；`record-merge` 在记录级按字段合并。这与 Supabase「一行一记录 + `updated_at` + RLS」天然贴合，服务端无需回放事件流。
- 第 17 节的 `SyncChange` **降级为客户端本地待上传队列的内部结构**，不作为云端事实源，云端不落地事件流表。
- 删除必须走 tombstone 传播，避免「一端删除、另一端重新上传导致复活」。tombstone 保留策略见技术方案。

**决策二：客户端只经 Supabase Edge Function 同步网关访问云端。**

- 所有同步读写经唯一的 Edge Function 端点，客户端（core）不直连 Postgres 数据表。
- 网关在服务端完成：敏感字段过滤、危险声明拒绝（`apiKey` / `token` / `filePath` 等，见第 8 节）、记录级冲突检测、快照触发。
- 这满足第 6.4 / 第 9 节「插件不能访问账号 token、不能直连云端接口」与第 15 节「服务端字段过滤」两条强约束，且贴合 Tabora「插件只经 core 访问平台能力」的架构边界。
- `service_role` 密钥只存在于 Edge Function 运行时，绝不进入扩展 / 前端 / 插件。
- 数据表仍开启 RLS 作为纵深防御（`TO authenticated using (account_id = (select auth.uid()))`，UPDATE 同时给 `USING` 和 `WITH CHECK`），即使未来出现直连路径也不越权。

**决策三：V1 加密语义为平台级加密 + 敏感字段永不上传，明确非 E2EE。**

- 对外表述统一为「传输 TLS + 存储 at-rest 加密（平台级）+ 敏感字段永不上传」，设置文案不得使用会被理解为端到端加密的措辞（如「所有数据已加密同步」）。
- 平台级 at-rest 加密意味着数据在库内对平台可读，因此敏感字段过滤（第 15 节）是隐私边界的主防线，加密只是存储层保护。
- 完整 E2EE 仍为非目标（第 4.2 / 第 20.2），作为后续独立版本重新评估。

## 4. 目标与非目标

### 4.1 目标

- 提供官方账号下的工作台云同步。
- 设置中心新增「数据同步」一级页。
- 支持官方账号注册、登录、退出登录和会话恢复。
- 未登录时保持本地工作台可用，同步能力关闭。
- 登录后自动注册当前设备为同步设备。
- 首次开启同步时让用户确认使用本机数据初始化云端，或与已有云端数据合并。
- 自动同步工作区结构、布局、插件实例和插件配置。
- 支持插件声明可同步业务集合。
- 自动排除密钥、本机路径、缓存和临时文件。
- 支持后台自动同步和手动立即同步。
- 支持记录级自动合并。
- 支持冲突收件箱处理不可自动合并的数据。
- 支持快照和恢复，降低误合并风险。
- 同步失败不阻塞工作台使用。

### 4.2 非目标

第一版不做：

- 完整端到端加密。
- 多账号共享工作区和协作编辑。
- 实时多人在线协同。
- 第三方 OAuth 登录。
- 复杂个人资料、头像、组织、团队和成员管理。
- 插件自建云同步协议。
- 插件读取云端账号令牌。
- 同步 AI API Key、模型密钥或本机安全存储。
- 同步本地文件原件、缓存目录、导出包和临时处理结果。
- Dashboard 同步状态卡片。
- 复杂容量计费和企业审计报表。

## 5. 用户角色

### 5.1 普通用户

普通用户需要知道：

- 我当前是否在本地模式。
- 如何注册或登录官方账号。
- 当前设备是否已经加入同步。
- 最近一次同步是否成功。
- 哪些内容会同步，哪些内容不会同步。
- 有没有冲突需要处理。
- 是否可以立即同步或从快照恢复。

普通用户不应该被迫理解记录版本、变更日志、合并算法或插件存储结构。

### 5.2 插件开发者

插件开发者只需要声明：

- 哪些业务集合可以同步。
- 每个集合的记录主键、更新时间和合并策略。
- 哪些字段必须排除。

插件开发者不处理账号、设备、云端请求、冲突收件箱、重试队列和加密存储。

### 5.3 平台维护者

平台维护者负责：

- 官方账号注册、登录态、会话恢复和设备注册。
- 同步队列和增量变更日志。
- 云端存储、服务端加密和敏感数据过滤。
- 自动合并和冲突收件箱。
- 快照恢复、失败重试和状态提示。

## 6. 官方账号体系

数据同步必须先有官方账号体系。用户未登录时，Tabora 继续作为本地工作台使用；账号页展示注册/登录入口、会话状态和设备注册状态，数据同步页只展示同步状态、同步范围、冲突入口和恢复入口，不展示伪造的已登录账号、已注册设备或云端状态。

### 6.1 账号入口

V1 使用邮箱验证码作为主要注册/登录方式。邮箱不存在时，用户完成验证码校验后创建新账号；邮箱已存在时，校验后登录已有账号。注册与登录可以共用同一表单，但界面必须让用户理解当前是在创建账号还是登录账号。

账号入口字段：

| 字段     | 必填       | 说明                                     |
| -------- | ---------- | ---------------------------------------- |
| 邮箱     | 是         | 官方账号主标识，用于登录、找回和同步绑定 |
| 验证码   | 是         | 发送到邮箱的 6 位验证码                  |
| 显示名称 | 注册时必填 | 默认可从邮箱前缀生成，用户可修改         |
| 协议确认 | 注册时必填 | 同意服务条款、隐私政策和数据同步说明     |

账号入口动作：

- 发送验证码。
- 登录已有账号。
- 创建新账号。
- 退出登录。

V1 不要求单独设计密码体系。若后续需要密码登录，应作为账号体系的扩展，不影响同步协议。

### 6.2 账号状态

| 状态             | 设置页表现                        | 同步能力     |
| ---------------- | --------------------------------- | ------------ |
| 本地模式         | 显示「未登录」，提供登录/注册入口 | 关闭         |
| 验证中           | 验证码发送后等待输入              | 关闭         |
| 已登录未开启同步 | 显示账号和当前设备待注册          | 等待用户开启 |
| 已同步           | 显示最近同步、设备和冲突状态      | 开启         |
| 会话过期         | 保留本地数据，提示重新登录        | 暂停         |
| 已退出登录       | 保留本机数据和本机快照            | 关闭         |

### 6.3 注册后的首次开启同步

账号登录或注册成功后，Core 自动注册当前设备，但不应立即无提示地上传本地数据。用户需要在设置页确认首次开启同步策略：

1. **使用本机数据初始化云端**：适用于新账号或云端没有工作台数据。
2. **合并云端已有数据**：适用于用户在另一台设备已经开启过同步。
3. **稍后开启**：保持已登录但同步关闭，本机继续本地使用。

如果云端已有数据，设置页必须先提示将进行合并，并在合并前生成本机快照。

### 6.4 会话与退出

- 登录会话由 core 管理，插件不可读取账号 token。
- 会话恢复失败时，工作台仍可本地使用，数据同步暂停。
- 退出登录不删除本地工作台数据。
- 退出登录后，当前设备停止同步；再次登录同一账号时重新校验设备资格。
- 用户可在设置页移除其他设备，不能直接用“移除设备”替代退出登录。

## 7. 同步对象

### 7.1 Core 默认同步对象

| 对象             | 说明                                                                |
| ---------------- | ------------------------------------------------------------------- |
| `Workspace`      | 当前活跃分组、分组列表、主题引用和背景配置                          |
| `Group`          | 分组名称、图标、默认布局、创建顺序和实例计数                        |
| `RegionState`    | 区域装配状态、布局扩展点和实例引用                                  |
| `PluginRecord`   | 插件身份、版本、来源、启用状态、manifest 摘要和已授权权限摘要       |
| `PluginInstance` | 实例 ID、所属分组、插件 ID、贡献 ID、区域、尺寸、网格位置和实例配置 |
| 插件设置         | 插件级默认配置和实例级配置                                          |

Core 对象由 Tabora 直接维护，不需要插件声明即可同步。

### 7.2 插件业务数据

插件业务数据只在插件 manifest 显式声明后同步。未声明的数据默认只留在本机。

典型可同步集合：

- 待办插件：任务、清单、完成状态、提醒策略。
- 今日重点插件：目标、步骤、复盘记录。
- 快捷入口插件：链接、分组、打开方式。
- 彩票查询插件：历史查询记录和用户保留的号码清单。

典型不应同步集合：

- 图片压缩插件的临时原图、缓存图、导出结果。
- 彩票 OCR 原图和一次性识别中间结果。
- 天气插件的请求缓存。
- 任何本机绝对路径。
- 任何 API Key、访问令牌和密钥材料。

## 8. 插件 manifest 同步声明

插件通过 manifest 声明可同步集合。Core 根据声明建立序列化、过滤、合并和冲突展示规则。

示例：

```json
{
  "id": "official.widget.todo",
  "permissions": ["storage"],
  "sync": {
    "collections": [
      {
        "name": "tasks",
        "recordKey": "id",
        "updatedAt": "updatedAt",
        "strategy": "record-merge",
        "excludeFields": ["localReminderId"]
      },
      {
        "name": "lists",
        "recordKey": "id",
        "updatedAt": "updatedAt",
        "strategy": "last-write-wins"
      }
    ]
  }
}
```

声明规则：

- `collections` 为空或缺失时，插件业务数据不参与同步。
- `recordKey` 必须稳定，不允许使用展示名称作为主键。
- `updatedAt` 必须由插件在写入时更新。
- `excludeFields` 用于排除设备私有字段或敏感字段。
- 第一版支持 `record-merge` 和 `last-write-wins` 两类策略。
- Core 可以拒绝明显危险的声明，例如同步 `apiKey`、`token`、`filePath` 字段。

## 9. Core 同步边界

Core 负责：

- 监听本地 IndexedDB 变更并生成同步变更。
- 合并同一账号下多设备的增量变更。
- 维护设备注册和设备状态。
- 过滤敏感字段和设备私有字段。
- 处理失败重试、离线队列和网络恢复。
- 生成自动快照。
- 将不可自动合并的记录投递到冲突收件箱。

插件负责：

- 使用本地 storage 写入业务数据。
- 在 manifest 中声明可同步集合。
- 为记录提供稳定 ID 和更新时间。
- 在 UI 中处理本地读写失败。

插件不能：

- 直接访问官方账号 token。
- 直接调用同步云端接口。
- 覆盖 core 的敏感字段过滤。
- 自行决定其他插件的数据同步策略。

> 实现约束（见 §3.1 决策二）：Core 也不直连云端数据表，而是经 Supabase Edge Function 同步网关读写。网关是敏感字段过滤、危险声明拒绝和冲突检测的唯一服务端执行点；`service_role` 密钥只存在于网关运行时。

## 11. 设备管理

用户登录官方账号后，当前设备自动注册为同步设备。

设备记录字段：

| 字段           | 说明                                     |
| -------------- | ---------------------------------------- |
| `deviceId`     | Core 生成的稳定设备 ID                   |
| `name`         | 用户可识别设备名                         |
| `type`         | macOS、Windows、iOS、Android、Browser 等 |
| `firstLoginAt` | 首次登录时间                             |
| `lastSyncAt`   | 最近同步时间                             |
| `status`       | 当前设备、在线、离线、已移除             |

设备移除规则：

- 用户可以在设置中移除其他设备。
- 移除设备只撤销后续同步资格，不删除云端数据。
- 当前设备不能直接从当前设备页移除，只能退出登录或关闭同步。
- 被移除设备再次登录需要重新注册。

## 12. 自动同步触发

默认启用后台自动同步。

触发时机：

- 本地数据变更后进入短延迟合并队列。
- 应用启动。
- 应用从后台回到前台。
- 网络从离线恢复。
- 官方账号登录成功。
- 用户点击「立即同步」。

队列规则：

- 多次本地写入先合并，再发起同步。
- 同步失败进入指数退避重试。
- 离线状态只记录待同步，不弹阻断式错误。
- 用户继续编辑时不被同步流程打断。

## 13. 冲突收件箱

Core 优先做记录级自动合并。只有同一记录在多设备上产生不可合并变化时，才进入冲突收件箱。

冲突条目包含：

- 插件或 core 对象名称。
- 记录名称或摘要。
- 本机设备、云端设备和时间。
- 本机版本摘要。
- 云端版本摘要。
- 建议处理方式。

用户操作：

- 保留本机。
- 保留云端。
- 手动合并。
- 暂时忽略。

处理完成后，Core 写入新的合并记录，并触发一次同步。

## 14. 快照与恢复

Core 在关键同步动作前自动生成快照：

- 首次开启同步前。
- 大批量云端合并前。
- 冲突批量处理前。
- 用户手动恢复前。

快照范围：

- Core 工作区结构。
- 插件实例和配置。
- 已声明可同步的插件业务数据。

快照不包含：

- API Key 和安全存储内容。
- 本机文件原件。
- 缓存、临时图片、导出结果。

恢复规则：

- 恢复前再次生成当前状态快照。
- 恢复成功后进入待同步状态。
- 恢复失败必须保留原状态并给出原因。

## 15. 安全与隐私

V1 使用平台级加密（传输 TLS + 存储 at-rest）和敏感数据排除，不承诺完整端到端加密（见 §3.1 决策三）。at-rest 加密只保护存储层，数据在库内对平台可读，因此敏感字段过滤是隐私边界的主防线。

安全规则：

- 官方账号 token 不暴露给插件。
- API Key 保持在本机安全存储，不进入同步。
- 本机绝对路径不进入同步。
- 缓存、临时图片、OCR 原图和导出结果不进入同步。
- 云端保存前经过服务端字段过滤（在 Edge Function 同步网关执行，见 §3.1 决策二），客户端过滤不可信、不作为唯一防线。
- 设置页明确展示同步范围和排除范围。

隐私提示必须写得具体：如实表述为「传输和存储均加密，敏感数据不上传」，不使用「所有数据已安全同步」「端到端加密」这类会被误解或夸大的承诺。

## 16. 失败状态

| 状态             | 表现                                       | 用户可做动作          |
| ---------------- | ------------------------------------------ | --------------------- |
| 未登录           | 本地模式，同步关闭，提示注册或登录官方账号 | 注册 / 登录           |
| 验证中           | 验证码已发送，等待用户输入                 | 输入验证码 / 重新发送 |
| 已登录未开启同步 | 当前设备待注册或已注册，但尚未开启同步     | 开启同步 / 稍后开启   |
| 已同步           | 最近同步时间可见                           | 立即同步              |
| 同步中           | 非阻塞 loading                             | 等待或继续使用        |
| 离线             | 保留本地变更，等待网络恢复                 | 稍后重试              |
| 冲突待处理       | 显示冲突数量和入口                         | 进入冲突收件箱        |
| 云端失败         | 显示失败原因摘要                           | 重试、稍后同步        |
| 设备已移除       | 当前设备失去同步资格                       | 重新登录              |

失败不得阻塞工作台打开和本地编辑。

## 17. 数据模型草案

```ts

  status: "active" | "disabled"

// SyncChange 是客户端本地待上传队列的内部结构（见 §3.1 决策一），

}
```

## 18. 典型流程

### 18.1 未登录进入数据同步

1. 用户在设置中心进入「数据同步」。
2. 数据同步页显示本地模式、同步范围、冲突入口和恢复入口。
3. 页面提示账号登录在「账号」页完成，不直接承载注册/登录表单。
4. 用户可以继续本地使用工作台，也可以切换到「账号」页注册或登录官方账号。

### 18.2 注册并首次开启同步

1. 用户输入邮箱和显示名称。
2. 用户点击「发送验证码」并输入邮箱验证码。
3. Core 创建官方账号和登录会话。
4. Core 自动注册当前设备。
5. 设置页让用户选择首次开启同步策略。
6. 用户选择「使用本机数据初始化云端」。
7. Core 创建开启前本地快照。
8. Core 上传 core 工作区结构和可同步插件数据。
9. 设置页显示最近同步时间、当前设备和同步范围。

### 18.3 登录已有账号并合并云端

1. 用户输入已有账号邮箱和验证码。
2. Core 恢复登录会话并注册当前设备。
3. Core 检测云端已有工作台数据。
4. 设置页提示用户选择「合并云端已有数据」或「稍后开启」。
5. 用户确认合并后，Core 先生成本机快照，再拉取云端数据。
6. 可自动合并的记录直接合并，不可合并的记录进入冲突收件箱。

### 18.4 多设备自动同步

1. 用户在 Mac 上新增待办卡片。
2. Core 记录 `PluginInstance` 和待办插件 `tasks` 变更。
3. 本地短延迟队列合并写入。
4. 网络可用时上传增量。
5. Windows 设备启动后拉取变更并合并。
6. Windows 工作台出现同一张待办卡片和任务数据。

### 18.5 冲突处理

1. 用户在 Mac 和 Windows 同时编辑同一条快捷入口。
2. Core 自动合并失败。
3. 冲突进入设置页「冲突收件箱」。
4. 用户查看两端摘要，选择合并。
5. Core 写入合并后记录并再次同步。

## 19. 验收标准

- 设置中心存在一级导航「数据同步」，位于「AI」和「插件」之间。
- 未登录时账号页展示注册/登录入口，数据同步页只展示本地模式和同步关闭状态。
- 注册/登录流程包含邮箱、验证码、显示名称和协议确认。
- 登录或注册成功后，当前设备自动注册，但首次开启同步需要用户确认策略。
- 数据同步页能展示状态、最近同步、冲突数、立即同步入口、同步范围、冲突入口和恢复入口。
- 设置页明确说明会同步的对象和不会同步的敏感对象。
- 账号页能展示当前设备注册状态，并提供其他设备管理入口。
- 冲突收件箱展示设备、时间、摘要和处理动作。
- 恢复快照入口说明快照范围和恢复后行为。
- 插件 manifest 有明确的同步声明格式。
- Core 与插件的同步职责边界清晰。
- 同步失败不阻塞本地工作台。
- PRD 不承诺 V1 完整端到端加密。

## 20. 风险与后续版本

### 20.1 风险

- 注册/登录和开启同步如果混在一个动作里，用户会误以为登录即上传本地数据。
- 插件声明字段不稳定会导致重复记录或错误合并；`updatedAt` 由插件维护，插件不打或打错时间戳会静默丢数据，Core 写入时应兜底盖一个可信时间戳。
- 本地 IndexedDB 结构演进会影响云端迁移；已通过 `SyncedRecord.schemaVersion`（§17）承接，迁移约定在技术方案细化。
- 服务端加密容易被误解为完整 E2EE；已通过 §3.1 决策三与 §15 文案约束消化。
- 自动同步如果过于频繁，会影响电量和网络。
- 数据同步页如果直接承载账号、设备、冲突列表和快照列表，会变成复杂控制台；V1 应只放轻量入口。
- Edge Function 网关是单一同步端点，其可用性直接影响同步；网关故障时客户端应回退到离线待同步队列，不阻塞本地工作台（见 §16）。
- 浏览器扩展（MV3）环境下 supabase-js 需注入自定义 storage adapter（如 `chrome.storage`）承载会话，否则 session 无法持久化；此为实现约束，技术方案需明确。

### 20.2 后续版本

- 密码登录、第三方 OAuth 和多因素认证。
- 端到端加密可作为独立版本重新评估。
- 支持工作区共享和多人协作。
- 支持企业设备审计和同步策略。
- 支持插件声明更细粒度的字段级合并器。
- 支持云端容量管理和历史版本浏览。

## 21. 结论

Tabora 数据同步 V1 应作为官方账号下的 core 能力实现，设置中心提供清晰可控的入口。用户未登录时保持本地模式，并在独立「账号」页通过邮箱验证码完成注册或登录；登录后自动注册当前设备，但首次开启同步仍需用户确认初始化或合并策略。「数据同步」页只承载同步状态、同步范围、冲突入口和恢复入口，避免变成复杂控制台。同步范围默认覆盖工作台结构和插件配置，插件业务数据必须显式声明后才进入云端。第一版以后台自动同步、敏感数据排除、记录级合并、冲突收件箱和快照恢复为核心，避免把同步做成工作台卡片或插件自建协议。
