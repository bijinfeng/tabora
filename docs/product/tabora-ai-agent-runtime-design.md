## 1. 设计目标

Tabora 的 AI 能力不定义为平台内置的单一聊天机器人，而定义为插件化工作台的 AI 基础设施。平台提供统一、安全、可配置的 AI runtime 和最小 agent 协议，插件负责具体 agent 体验。

这个方向解决两个问题：

- 多个 AI 插件不需要重复实现模型配置、流式响应、工具调用、上下文读取、权限授权、错误处理和用量状态。
- AI 能力仍然遵守 Tabora 的插件优先原则：平台保留通用运行机制，具体业务能力仍由官方或第三方插件贡献。

核心定位：

> Tabora 不是做一个 AI 聊天新标签页，而是让每个插件都能低成本获得一致、安全、可配置的 agent 能力。

### 1.1 当前技术选型

P0 阶段采用 **Tabora 自有协议 + Vercel AI SDK adapter**：

- `@tabora/plugin-api` 定义插件可见的 AI 请求、流式响应、工具声明、工具审批和 `ai` 权限类型。
- `@tabora/platform-kernel` 只负责在 `PluginRuntimeContext` 中按权限暴露 `context.ai`，不直接依赖 Vercel AI SDK。
- `@tabora/ai-runtime` 作为平台基础设施包，基于 Vercel AI SDK 的 `generateText` / `streamText` 实现默认 adapter。
- 插件只依赖 Tabora 的 `context.ai`，不直接依赖 Vercel AI SDK，也不保存模型密钥。

这条路线保留后续替换 Pi、LangGraph、OpenAI Agents SDK 或服务端 runner 的空间，因为插件协议不绑定具体第三方框架。

## 2. 产品原则

### 2.1 平台提供基础设施

平台负责：

- 模型供应商与默认模型配置。
- AI 请求网关与流式响应。
- Agent 工具调用协议。
- 工作区上下文摘要。
- 用户确认后执行。
- AI 权限、用量、错误和状态展示。

平台不负责：

- 把所有 AI 场景写成一个内置超级助手。
- 直接承载具体业务 agent 的长期逻辑。
- 绕过插件协议读写插件私有数据。
- 让 AI 绕过现有权限桥执行外部打开、剪贴板、本地文件或 workspace 写入。

### 2.2 插件提供具体 agent

插件负责：

- 便签整理 agent。
- 待办拆解 agent。
- 搜索研究 agent。
- 图片处理 agent。
- 股票、彩票、天气等领域 agent。
- 未来第三方自定义 agent。

插件可以扮演三种角色：

- Agent Consumer：使用平台 AI 能力，例如便签总结。
- Agent Provider：提供可被用户调用的 agent，例如待办规划助手。
- Tool Provider：提供可被 agent 调用的工具，例如创建待办项、添加快捷入口。

MVP 不需要一次性做满三种角色。第一阶段优先支持 Agent Consumer 和 Tool Provider 的最小版本，再用官方示范 agent 验证 Provider 形态。

### 2.3 AI 可以规划，平台负责执行边界

AI 可以生成回答、建议和工具调用草案。涉及写入、外部打开、跨插件读取完整内容、剪贴板、本地文件等敏感动作时，必须由宿主展示确认并通过权限桥执行。

第一阶段采用“确认后执行”：

- AI 可以提出操作计划。
- 用户确认后平台执行。
- 不支持授权后连续自动执行多步任务。

## 3. MVP 范围

### 3.1 包含范围

第一阶段交付 AI Runtime 和最小 Agent 协议：

- 支持配置模型供应商、API key、base URL、默认文本模型、默认视觉模型。
- 提供 AI Gateway 状态显示和测试连接。
- 提供统一调用能力，例如 `ai.generate` 和 `ai.stream`。
- 提供标准化工作区摘要：当前分组、布局、widget 实例列表、插件名称、贡献类型、卡片标题和基础状态。
- 插件可以声明 agent action 或工具调用能力。
- Agent 提出的工作台修改、插件数据写入、外部打开等动作必须经过用户确认。
- 设置中心新增 AI 面板：模型配置、网关状态、插件 AI 授权。
- 提供一个官方示范 agent 插件，用来验证协议。
- 改造 1-2 个已有官方插件作为 AI consumer 示例，例如便签总结、待办拆解。

### 3.2 不包含范围

第一阶段不做：

- 自动连续执行多步任务。
- 跨插件读取完整私有内容。
- 长期记忆系统。
- AI 生成插件。
- 远程 agent 市场。
- 云同步和账号级 AI 配额。
- 无确认的 workspace 写入。

## 4. 用户场景

### 4.1 工作台助手

用户输入“帮我把今天要做的事整理一下”。AI 基于当前工作区摘要生成建议，例如创建待办、调整卡片、打开相关搜索。涉及写入或外部打开时，宿主展示确认卡片。

### 4.2 插件内助手

便签、待办、图片处理等插件可以声明自己的 agent 能力：

- 便签：总结当前便签、提取待办。
- 待办：拆解任务、生成步骤、估算优先级。
- 图片处理：解释当前图片处理结果或建议压缩参数。

### 4.3 命令搜索增强

`⌘K` 不只匹配命令、搜索源和卡片，也可以出现 agent action。例如输入“整理便签”，出现“让便签助手整理当前便签”。

### 4.4 设置与授权

用户在设置中心配置模型，检查网关状态，并决定哪些插件可以调用 AI、读取哪些上下文、请求哪些工具。

## 5. 核心体验流程

1. 用户从 `⌘K`、某个 widget、展开视图、设置面板或官方 AI 助手插件发起请求。
2. 插件调用平台 AI Runtime，并声明本次请求需要的上下文和工具。
3. 平台生成工作区摘要，检查插件 AI 授权。
4. AI 返回文本、结构化建议或工具调用草案。
5. 如果只是回答或总结，直接展示结果。
6. 如果要写入工作区、写入插件数据、打开外部 URL、访问剪贴板或本地文件，平台展示确认。
7. 用户确认后，平台通过已有 host/runtime 能力执行。
8. 执行结果以 Toast 或结果块反馈。
9. 失败时只影响本次 agent 会话，不影响插件或整个工作台。

## 6. 平台能力

### 6.1 AI Gateway

AI Gateway 统一处理：

- 模型供应商。
- API key。
- base URL。
- 默认模型。
- 请求超时。
- 流式响应。
- 错误归一化。

当前实现的默认底层 adapter 放在 `@tabora/ai-runtime`，基于 Vercel AI SDK。P0 先覆盖文本生成、流式文本和受控工具执行；provider 配置、密钥存储、连接测试和用量统计仍由后续设置中心与宿主能力补齐。

插件不直接保存模型密钥，也不读取明文 API key。

### 6.2 AI Runtime Context

平台向插件暴露受控 AI API。产品层面建议包含：

- 生成文本。
- 流式生成。
- 运行 agent 请求。
- 获取工作区上下文摘要。

当前最小协议已收敛为 `context.ai.generate`、`context.ai.stream`、可选 `context.ai.requestToolApproval` 和可选 `context.ai.getWorkspaceContext`。插件必须声明 `ai` 权限后才会获得对应 bridge。

### 6.3 Agent Tool Registry

插件可以声明可被 agent 请求的工具。平台负责：

- 收集工具声明。
- 检查工具所属插件和权限。
- 将 AI 输出转换成工具调用草案。
- 在执行敏感工具前展示确认。
- 通过宿主能力执行已确认动作。

工具不应绕过平台直接执行危险动作。

### 6.4 Workspace Context Provider

平台提供标准化工作区摘要，默认不泄露插件私有内容：

- 当前工作区或分组名称。
- 当前布局。
- widget 实例列表。
- 插件名称。
- contribution 类型。
- 卡片标题。
- 基础状态摘要。
- 用户当前触发来源。

完整插件私有内容必须由插件显式提供，并经过用户授权。

### 6.5 AI Permission Bridge

在现有权限模型上增加 AI 相关能力：

- 调用模型：`{ type: "ai", access: ["generate"] }`，允许插件使用平台 AI Gateway。
- 读取上下文：`{ type: "ai", access: ["context"] }`，允许读取当前触发上下文、工作区摘要或插件显式提供的摘要。
- 请求工具：`{ type: "ai", access: ["tools"] }`，允许插件请求 agent 工具调用审批。

执行动作继续复用现有或后续扩展权限：

- `workspace` write。
- `external-open`。
- `clipboard`。
- `local-file`。
- `network`。

AI 不应该成为绕过这些权限的捷径。

### 6.6 Agent Confirmation Host

所有写入类、外部打开类和跨插件操作由宿主统一展示确认。确认内容必须说明：

- 哪个插件发起。
- 哪个 agent 发起。
- 将执行什么动作。
- 会影响哪些数据。
- 是否调用外部能力。

插件不应自行手搓高风险确认弹窗。

## 7. 插件协议方向

建议后续在 manifest contribution 中增加或扩展以下能力：

- `agentProviders`：声明插件提供哪些 agent。
- `agentTools`：声明 agent 可请求哪些工具。
- `agentContext`：声明插件愿意提供哪些上下文摘要。
- `commands`：让 agent action 可以进入 `⌘K`。
- `settingsPanels`：继续用于插件自己的 AI 配置页。

MVP 可先不引入完整协议，只交付最小闭环：

- 插件声明自己需要 AI 能力。
- 插件可以调用平台 AI Gateway。
- 插件可以提供有限工具声明。
- 工具执行前由宿主确认。

## 8. 信息架构

### 8.1 设置中心 / AI

新增一级设置面板“AI”，承载模型、网关、授权和用量状态。它是 AI 能力控制中心，不是聊天入口。

建议分区：

- 网关状态：当前 provider、连接状态、最后一次错误、测试连接。
- 模型配置：默认文本模型、默认视觉模型、基础生成参数。
- 插件授权：哪些插件允许调用 AI、读取上下文、请求工具调用。
- 用量与安全：本地请求次数、失败次数、可选 token 估算、敏感操作确认策略。

### 8.2 命令搜索 / Agent Action

`⌘K` 中展示 agent action。Agent action 应和现有 command/search 结果一样可键盘访问、可分组、可解释来源。

### 8.3 插件内容区

具体插件在自己的 widget、展开视图或设置面板中放置 AI 操作。例如便签卡片的“整理”、待办卡片的“拆解”。这些操作必须调用平台 AI Runtime。

## 9. 官方示范插件

建议新增官方插件 `official.ai.assistant`。

职责：

- 提供“工作台助手”入口。
- 读取工作区摘要。
- 生成建议。
- 提出待确认操作。
- 将 agent action 注册到 `⌘K`。
- 展示 AI Runtime 未配置、未授权、请求失败等状态。

同时改造 1-2 个已有插件作为 AI consumer 示例：

- 便签：总结、提取待办。
- 待办：拆解任务、生成步骤。

示范插件不应该垄断所有 AI 体验。它的主要目标是验证平台协议，帮助未来第三方 agent 插件复用基础设施。

## 10. 权限与安全

### 10.1 权限分层

AI 权限分为四层：

1. 调用模型：插件是否可以使用平台 AI Gateway。
2. 读取上下文：当前触发上下文、工作区摘要、插件私有内容分级授权。
3. 请求工具：插件是否可以让 AI 生成工具调用草案。
4. 执行动作：继续复用 workspace、external-open、clipboard、local-file、network 等权限。

### 10.2 安全原则

- 默认不让插件无感使用 AI；首次调用时需要明确授权或在设置中开启。
- AI 只能请求敏感动作，不能直接执行。
- 跨插件读取完整内容必须显式授权。
- API key 由平台保存，插件不可读取明文。
- AI 错误只影响本次请求，不导致插件崩溃。
- Agent 工具调用应有结构化本地日志，至少可查看最近记录。

## 11. 失败状态

MVP 必须覆盖：

- 未配置模型：提示去 AI 设置。
- Provider 测试失败：显示具体错误摘要，不暴露密钥。
- 插件未授权：展示授权入口。
- 上下文权限不足：说明缺少哪类上下文。
- 工具调用需要确认：展示确认卡片。
- 工具执行失败：结果块显示失败原因，并保留原始建议。
- AI 返回不可解析工具调用：降级为纯文本建议。
- 流式响应中断：保留已生成内容，并允许重试。

## 12. 验收标准

第一阶段完成后应满足：

- 用户可以在设置中心配置 AI provider、API key、默认文本模型和默认视觉模型。
- 设置中心能显示 AI Gateway 状态，并支持测试连接。
- 插件可以通过平台 Runtime 调用 AI，而不是自己保存密钥或直接请求模型。
- 至少一个官方示范 agent 可以读取工作区摘要并生成建议。
- 至少一个已有插件可以作为 AI consumer 调用平台能力，例如便签总结或待办拆解。
- AI 发起写入、外部打开、跨插件操作时必须展示用户确认。
- 用户可以查看和管理插件 AI 授权。
- 未配置、未授权、请求失败、流式中断、工具调用失败都有清晰降级。
- AI 能力不破坏现有插件边界：业务能力仍在插件，平台只提供通用 AI Runtime、权限和宿主确认。

## 13. 分阶段路线图

### P0：AI 基础设施闭环

- AI 协议类型与 manifest `ai` 权限。
- `@tabora/ai-runtime` Vercel AI SDK adapter。
- `context.ai.generate` 和 `context.ai.stream`。
- 敏感工具执行前的审批回调。
- 未配置和 provider 失败的归一化错误。

P0 尚未完成的宿主能力：

- AI provider / API key 设置 UI。
- AI Gateway 状态与测试连接。
- 插件 AI 授权管理面板。
- 工作区上下文摘要注入。

### P1：Agent 最小协议

- 工作区上下文摘要。
- Agent action 声明。
- 工具调用草案。
- 用户确认后执行。
- 官方工作台助手插件。

### P2：插件 AI 示例

- 便签总结和提取待办。
- 待办拆解和步骤生成。
- `⌘K` 中展示 agent action。
- 插件 AI 授权面板。

### P3：增强能力

- 视觉模型支持图片类插件。
- 更细的上下文授权。
- 本地 agent 调用日志。
- 多 agent 插件共存。
- 用量统计和请求限制。

## 14. 开放问题

- AI Gateway 的密钥保存位置和加密策略是否需要区分 web、extension 和未来 desktop。
- AI provider 是否允许插件声明推荐模型，还是只由用户全局选择。
- 工作区摘要的字段粒度如何标准化，避免泄露插件私有内容。
- Agent 工具调用日志保留多久，是否支持用户清空。
- 插件 AI 授权是按插件、按 agent、按工具，还是三者组合。
- 视觉模型能力是否进入第一阶段，还是只保留默认视觉模型配置入口。
