## 1. 产品概述

Tabora 是一个插件优先的个人工作台新标签页产品。它不是一个固定写死搜索、壁纸和卡片的新标签页，而是一个由插件装配出来的模块仪表盘型个人工作台平台。

平台本身只提供插件运行内核、扩展点协议、宿主容器、存储、权限、事件、故障恢复机制、全局可达性保障和最小安全布局。用户实际看到的布局结构、搜索形式、rail 样式、卡片排列方式、背景、主题、卡片内容、弹窗和设置面板，都来自官方或未来第三方插件。

核心哲学：**一切皆插件，包括布局**。平台不预设"正确"的布局——它只定义布局协议和全局约束，让不同布局插件在约束内自由表达。

## 2. 产品定位

Tabora 面向希望把浏览器新标签页变成个人工作入口的用户。它强调可组合、可扩展和可持续演进，而不是只提供单一固定页面。

核心定位：

- 模块仪表盘型个人工作台，而不是纯极简起始页或单纯搜索页。
- **布局本身也是插件**。官方默认布局（仪表盘式）基于创建者的个人习惯设计；其他用户可以通过安装不同布局插件获得完全不同的工作台结构。平台不预设"正确"的布局——它只保证布局协议和全局可达性约束。
- 默认体验由官方内置插件组合出来。
- 平台不硬编码具体业务能力（包括布局结构）。
- 用户可以按自己的工作方式添加、删除、调整和配置插件实例——包括切换整个工作台布局。
- 插件生态可以逐步从官方插件扩展到本地可信插件，再扩展到第三方插件市场。
- **MVP 必须提供至少两种不同结构范式的布局插件**，以验证布局协议的通用性和插件平台的扩展性。

两种官方布局插件（MVP 标配）：

- **Dashboard（仪表盘式）**：左侧轻导航 rail 承载主页、添加卡片、插件和设置等工作台级入口；顶部命令搜索承载网页搜索、搜索源选择和快捷建议；主区域使用可拖拽、可调整尺寸、可纵向滚动的 4 列自适应网格。适合习惯多卡片并行的用户。
- **Stream（流式）**：无固定 rail，命令搜索通过 ⌘K 浮层唤起，卡片纵向堆叠（桌面双列、移动端单列），页面以大背景起始，向下滚动呈现 widget 内容。适合专注型、移动端和偏好简洁界面的用户。

布局设计原则（适用所有布局插件）：

- 布局只决定"卡片怎么排列"和"搜索/导航怎么呈现"——不改变卡片内容，不干涉插件数据。
- 切换布局时保留所有插件实例数据；新布局中无法匹配区域的实例进入"待放置"状态。
- 所有布局必须满足 2.1.3 节「全局可达性约束」。

不采用的方向：

- 不把 MVP 做成只有搜索框和快捷链接的新标签页入口。
- 不在 MVP 直接做复杂“今日流 / 日程驱动”产品，避免过早进入任务管理系统。

最高产品原则：

> 任何具体业务能力默认优先放进插件；平台只保留通用运行机制。

补充约束原则：

> 插件在约束框架内自由——平台定义边界和契约，插件在边界内自主。自由不意味着可以绕过权限、打破一致性或让用户迷失入口。

## 2.1 插件自由度与平台约束

Tabora 的插件化不是"什么都能做"的无政府状态。平台为每一类插件定义清晰的**自由度**（插件能自主决定什么）和**约束**（插件必须遵守什么）。这种平衡确保：即使未来有数十个第三方布局和数百个 widget，用户体验仍然一致、可预测、不混乱。

### 2.1.1 布局插件 (layout) 的约束

布局插件是自由度和责任最大的插件类型——它决定工作台的整体结构。

| 自由度                                              | 约束                                                                                                 |
| --------------------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| 定义任意数量和类型的区域 (region)                   | 每个区域必须声明接受的扩展点类型（如 `widget`、`search`）                                            |
| 决定区域的排列方式（网格、流式、列式、看板等）      | 不能重复声明已被其他区域占用的扩展点                                                                 |
| 决定 rail 的存在形式（固定侧栏/顶部工具条/无 rail） | 如果布局取消了常驻 rail，必须提供替代入口（如 ⌘K 或顶部菜单）让用户可达：添加卡片、插件管理、设置    |
| 决定搜索的呈现方式（常驻栏/浮层唤起/侧栏内嵌）      | 搜索入口必须始终可达——每个布局必须至少注册一个 `search` 扩展点区域，或通过平台级 ⌘K 快捷键保证可唤起 |
| 决定响应式折叠策略                                  | 任何断点下都不允许出现横向滚动                                                                       |
| 决定区域的嵌套深度                                  | 卡片最小宽度不得低于 180px（保证可读性），不受布局类型影响                                           |

### 2.1.2 Widget 插件的约束

Widget 是用户接触最频繁的插件类型。

| 自由度                         | 约束                                                                  |
| ------------------------------ | --------------------------------------------------------------------- |
| 决定卡片内部内容的渲染         | 必须声明 `supportedSizes`（至少一个），实际尺寸在声明范围内由用户选择 |
| 决定卡片内部的交互逻辑         | 内容区不得撑破卡片外壳——过长时内部滚动或截断                          |
| 可以声明 modal/fullscreen 视图 | 弹窗/全屏只能通过 runtime context 请求宿主打开，不能自行创建全局浮层  |
| 可以读写插件私有数据           | 不得直接读写宿主 workspace 状态或其他插件的数据                       |
| 可以使用任意 lucide 图标       | 不得使用 emoji 作为功能图标                                           |
| 可以使用自定义样式             | 主文本/表面色/边框色必须走 theme token，不得硬编码大面积色值          |

### 2.1.3 全局可达性约束（所有布局必须满足）

无论活跃布局是哪一种，以下入口和功能必须在**任何布局下可达**。这是平台级的体验底线，也是布局插件的强制合规项。

| 入口/功能    | 可达性要求                                                                                                                             |
| ------------ | -------------------------------------------------------------------------------------------------------------------------------------- |
| 命令搜索     | 每个布局必须提供搜索通道：常驻搜索栏、⌘K 浮层、或侧栏内嵌搜索。⌘K 由平台注册为全局快捷键，即使布局未显式放置搜索区域也可触发浮层搜索。 |
| 添加卡片     | 可通过 rail、顶部工具条、⌘K 命令或设置中心进入。                                                                                       |
| 插件管理     | 可通过 rail、设置中心或 ⌘K 命令进入。                                                                                                  |
| 设置中心     | 可通过 rail、顶部工具条或 ⌘K 命令打开，所有布局下设定面板由宿主 draw 统一承载。                                                        |
| 主题切换     | 可在设置中心完成，或通过 ⌘K 命令快速切换。                                                                                             |
| 卡片尺寸调整 | 每个 widget 卡片在 hover 或右键时提供尺寸菜单入口。                                                                                    |

### 2.1.4 主题与视觉一致性约束

| 约束       | 说明                                                                                         |
| ---------- | -------------------------------------------------------------------------------------------- |
| Token 强制 | 所有插件 UI 中的主文本、次级文本、表面色、边框色、强调色必须使用主题 token，不得硬编码色值。 |
| 暗色兼容   | 每个插件贡献的视图必须在平台当前主题（明亮/暗色）下保持可读。                                |
| 卡片稳定   | hover、focus、拖拽、loading 等状态不得改变卡片外部尺寸。                                     |
| 控件一致   | 插件内容区的按钮、输入框、选择器等优先使用 `@tabora/ui` 基础组件。                           |

### 2.1.5 错误隔离约束

| 约束         | 说明                                                                                    |
| ------------ | --------------------------------------------------------------------------------------- |
| Widget 失败  | 只显示该实例错误卡片，不影响同布局下其他卡片。                                          |
| 布局插件失败 | 回退到平台内置的**最小安全布局**（单列流式 + ⌘K 搜索 + 设置入口），保证用户不面对白屏。 |
| 搜索插件失败 | 搜索区域显示不可用占位，⌘K 仍然可唤起设置/命令。                                        |
| 主题插件失败 | 回退到平台内置安全 token。                                                              |
| 设置面板失败 | 仅该面板区域显示错误，设置中心其他面板正常。                                            |

## 3. 背景与问题

传统新标签页产品通常把搜索框、快捷链接、天气、便签、壁纸和主题做成固定功能。这种模式的早期体验很直接，但后续会遇到几个问题：

- 功能增长后，平台代码会越来越耦合。
- 用户不能按插件维度自由组合能力。
- 第三方生态很难接入。
- 主题、背景、搜索源、卡片和设置面板之间缺少统一扩展协议。
- 单个功能出错时，容易影响整个页面。

Tabora 的产品假设是：新标签页应该是一个可装配的个人工作台，而不是一组固定组件。只要扩展点、运行时、权限和持久化边界设计清楚，默认体验和未来生态可以在同一套机制上演进。

## 4. 目标与非目标

### 4.1 产品目标

V1 MVP 要证明：

- 插件内核可以装配出完整产品体验。
- **布局本身可以是插件，且不同结构范式的布局可以在同一平台上共存。**
- **平台约束体系（区域协议、全局可达性、响应式安全、视觉一致性）可以有效防止插件自由导致的体验碎片化。**
- 官方插件可以组成默认工作台。
- 用户可以添加、删除、调整尺寸、配置和展开插件实例。
- 状态可以跨会话持久化。
- 插件渲染错误不会拖垮整个工作台。
- 外部能力调用通过权限桥进入宿主，不由插件直接越权执行。
- 官方插件 UI 使用统一基础组件和主题 token，避免每个插件重复实现控件样式。

### 4.2 非目标

MVP 不做：

- 第三方远程插件市场。
- 不可信远程插件沙箱运行时。
- 在线插件安装和自动升级。
- 云同步和账号系统。
- 团队或共享工作区。
- 插件评论、评分和审核系统。
- 完整插件开发者工具。
- AI 生成插件。
- 复杂 WebGL 背景编辑器。
- 深度移动端应用适配。

## 5. 目标用户

### 5.1 核心用户

个人效率型用户：

- 每天频繁打开浏览器新标签页。
- 希望新标签页承载搜索、快捷入口、便签、待办、天气、RSS、背景和主题。
- 需要按自己的习惯组合工作区。

### 5.2 次级用户

插件开发者和高级用户：

- 希望用插件扩展自己的新标签页。
- 需要稳定的 manifest、contribution、runtime context 和 storage 协议。
- 未来可能发布本地插件或第三方插件。

### 5.3 内部用户

产品和研发团队：

- 需要快速扩展官方插件能力。
- 需要保持平台代码和业务插件解耦。
- 需要用同一套协议支撑 playground、浏览器扩展和未来其他 shell。

## 6. 用户价值

用户价值：

- 可以把新标签页变成自己的工作入口。
- 可以自由添加多个同类卡片实例。
- 可以按卡片支持的尺寸调整布局。
- 可以切换主题和背景。
- 可以展开插件提供的弹窗或全屏视图。
- 关闭并重新打开后，工作区状态完整恢复。

平台价值：

- 新能力通过插件扩展，不污染平台核心。
- 官方插件和未来第三方插件共用同一运行协议。
- 安全、权限、持久化和故障隔离从 MVP 开始内建。

## 7. MVP 范围

### 7.1 包含范围

MVP 包含：

- 插件发现、manifest 校验、激活、registry、runtime context。
- 扩展点：`layout`、`widget`、`search`、`search-provider`、`background-provider`、`background-renderer`、`theme`、`settings-panel`。
- **至少两种 `layout` 插件的官方实现**，验证布局协议在不同结构范式下的扩展性和可靠性。
- **布局切换能力**：用户可在设置中选择活跃布局插件，切换后保留实例数据并重新渲染壳体。
- **最小安全布局**：当活跃布局插件渲染失败时，平台回退到一个内置安全布局（单列流式 + ⌘K 搜索 + 设置入口），保证用户不面对白屏。
- 官方内置插件包。
- `@tabora/ui` 基础组件包，为官方插件和未来第三方插件提供一致的内容区控件。
- 左侧轻导航 rail + 顶部命令搜索 + 主网格工作台默认布局。
- Widget 多实例和多尺寸支持。
- 拖拽排序和网格状态持久化。
- 卡片过多时的纵向溢出策略，保证主网格可滚动、可排序、可调整尺寸。
- 弹窗和全屏宿主。
- 通过 token 切换主题。
- 切换背景来源和背景渲染器。
- 搜索源启用、禁用和默认搜索源。
- 轻量设置中心：宿主提供 settings host，聚合插件、外观、搜索三个 settings-panel contribution；插件启停、权限审计、导入导出后移。
- IndexedDB 本地持久化。
- 插件错误边界和安全回退状态。
- `external-open` 最小权限桥。

### 7.2 官方内置插件

MVP 推荐官方插件：

- `official.layout.workbench-dashboard`：仪表盘式布局。左侧 56px rail + 顶部 48px 命令搜索栏 + 4 列自适应网格。搜索栏常驻可见，rail 提供主页/添加卡片/插件/设置入口。响应式折叠：平板 3 列，移动端 rail 收为底部工具条、单列堆叠。
- `official.layout.workbench-stream`：流式布局。无固定 rail，顶部提供可折叠的轻工具条（仅含设置入口），命令搜索通过 ⌘K 浮层唤起。卡片在桌面端以双列流式排列、移动端单列。页面以大背景起始，向下滚动呈现 widget。响应式折叠：平板双列，移动端单列。桌面端工具条可设置为自动隐藏以获得更沉浸的视野。
- `official.search.command-bar`：提供命令搜索 UI 和搜索编排，MVP 先覆盖网页搜索、搜索源选择和快捷建议。
- `official.search-providers.basic`：提供 Google、Bing、百度、DuckDuckGo、GitHub 等搜索源。
- `official.background.providers.basic`：提供基础背景来源。
- `official.background.renderers.basic`：提供基础背景渲染。
- `official.theme.default-pack`：提供明亮和暗色等主题 token。
- `official.widgets.today-focus`：提供“今日重点”卡片，用最轻量的方式建立个人工作台心智。
- `official.widgets.quick-links`：提供快捷入口卡片，验证外部打开和实例配置。
- `official.widgets.notes`：提供便签卡片和弹窗展开，验证插件私有数据和宿主 modal。
- `official.widgets.todo`：提供待办卡片，验证交互型 widget 和持久化。
- `official.widgets.weather`：提供天气摘要卡片，MVP 可作为可添加候选，不一定默认进入首屏。
- `official.plugin-manager`：提供插件设置面板，MVP 展示插件状态、贡献能力和权限摘要，默认从设置中心进入，后续承载启用、禁用和版本管理。
- `official.settings.workspace`：提供 MVP 轻量设置中心，聚合插件贡献的设置面板，并组织外观、搜索和基础工作区设置。

## 8. 核心功能需求

### 8.1 默认工作台与多布局

用户首次打开应用时，应看到一个完整可用的默认工作台。MVP 以仪表盘式布局为默认，用户可在设置中切换到流式布局。

要求：

- 默认工作台由官方插件装配出来。默认使用仪表盘式布局插件。
- 仪表盘式布局包含左侧轻导航 rail、顶部命令搜索区域和下方主网格区域。
- 流式布局包含可折叠顶部工具条、⌘K 浮层搜索和双列卡片流。
- 两种布局下，默认都包含基础 widget 实例：今日重点、快捷入口、便签和待办。
- 天气和插件状态可作为可添加候选或设置入口，不要求默认进入首屏。
- 默认首屏只保证命令搜索和 3-4 个核心卡片优先露出，不强求所有官方卡片都在一屏内完整显示。
- 默认主题和背景来自插件 contribution。
- 平台代码不硬编码具体业务功能——包括具体布局结构。

### 8.2 布局切换

用户可以在设置中切换活跃的布局插件。切换时保留所有插件实例数据，由新布局重新编排卡片。

要求：

- 设置中心的外观面板列出所有已启用的 `layout` contribution。
- 用户选择新布局后，平台切换到该布局插件，重新渲染工作台壳体。
- 切换布局时保留所有 `PluginInstance` 数据（卡片内容、配置、尺寸不变）。
- 新布局的区域结构与旧布局不同时，尝试按区域 ID 匹配；未匹配到有效区域的实例进入「待放置」状态，用户在卡片列表中可见并可拖入新布局的有效区域。
- 布局选择通过 workspace 状态持久化，刷新后恢复。
- 当活跃布局插件渲染失败时，平台回退到内置**最小安全布局**（单列流式 + ⌘K 搜索 + 设置入口），确保用户不面对白屏。
- 布局切换期间，平台显示短暂过渡状态（骨架屏或空壳），避免闪烁。

### 8.3 插件发现与激活

平台启动时完成插件装配。

要求：

- 扫描内置插件和已安装插件。
- 校验 manifest。
- 建立插件索引。
- 激活已启用插件。
- 插件通过 runtime context 注册视图和能力。

### 8.4 Widget 多实例

用户可以添加同一个 widget 插件的多个实例。

要求：

- 添加面板列出已启用的 `widget` contribution。
- 用户选择 widget 后，平台创建新的 `PluginInstance`。
- 每个实例有独立 ID、尺寸、区域、位置和配置。
- 删除实例只影响该实例，不删除插件本体。

### 8.5 Widget 尺寸调整

用户可以从插件声明支持的尺寸中选择实例尺寸。

要求：

- 平台读取 `supportedSizes`。
- 可选尺寸包括 `S`、`M`、`L`、`XL` 的子集。
- 尺寸变化更新实例状态。
- 当前布局把语义尺寸映射到网格跨度。
- 尺寸状态跨会话恢复。

### 8.6 拖拽排序与布局持久化

用户可以调整工作台卡片顺序。

要求：

- 支持在主网格区域拖拽排序。
- 排序结果写入 `PluginInstance.grid`。
- 关闭并重新打开后恢复位置。
- 布局数据和插件业务数据分开存储。

### 8.7 卡片溢出与首屏策略

当用户添加的卡片超过一屏可容纳数量时，MVP 不应压缩所有卡片来强行塞进一屏，而应保持卡片可读性和布局稳定。

要求：

- 主网格允许纵向扩展，超出视口时页面纵向滚动。
- 不使用横向滚动作为默认溢出方案。
- 新增卡片默认追加到主网格末尾，不打乱用户已有卡片顺序。
- 用户可以通过拖拽排序把高频卡片移到上方。
- 用户可以通过尺寸调整把低优先级卡片压缩为更小尺寸。
- 默认工作台只放入少量核心卡片；低频官方卡片进入添加面板或设置中心。
- 移动端采用单列堆叠，保持语义尺寸状态，但布局可按移动端规则折叠。
- MVP 不做强制分页、复杂分组、自动收纳、多 workspace 和卡片搜索，这些能力进入后续版本。

### 8.8 轻导航 Rail

Rail 由布局插件定义其存在形式和交互语义。不同布局可以有不同的 rail 实现——甚至没有 rail。平台负责提供访问入口的通用机制，不强制 rail 的外观。

要求：

- Rail 的存在形式由布局插件决定：仪表盘式布局使用 56-64px 固定左侧 rail；流式布局使用可折叠顶部轻工具条。
- 无论 rail 形式如何，以下入口在所有布局下必须可达（通过 rail、顶部工具条、⌘K 命令或设置中心）：添加卡片、插件管理、设置、主题切换。
- 桌面端 rail（如果有）宽度控制在 56-64px，只放图标或极短标签。
- 移动端 rail 可折叠为底部工具条或顶部紧凑操作区，不能造成横向滚动。
- Rail 不直接承载具体业务内容，具体内容仍通过 widget、search 或 settings-panel 插件呈现。

### 8.9 命令搜索

命令搜索是 Tabora 的核心交互入口。它在所有布局下必须可达——形式可以是常驻搜索栏（仪表盘式）或 ⌘K 浮层（流式）。

要求：

- 搜索 UI 来自 `search` 插件。
- 搜索源来自 `search-provider` contribution。
- 仪表盘式布局中搜索栏常驻于主区域顶部；流式布局中搜索通过 ⌘K 浮层唤起。
- ⌘K 由平台注册为全局快捷键——即使布局未显式放置搜索区域，用户也可通过快捷键唤起浮层搜索。
- MVP 支持搜索源选择、查询提交和基础快捷建议。
- MVP 可以展示快捷入口建议，但不要求搜索本地卡片内容、插件命令或历史记录（后续版本扩展）。
- 打开外部 URL 必须通过 `context.permissions.openExternal`。

### 8.10 主题与背景

用户可以切换主题和背景。

要求：

- 主题来自 `theme` contribution。
- 主题输出 token set，平台应用为 CSS custom properties。
- 背景来源来自 `background-provider` contribution。
- 背景渲染来自 `background-renderer` contribution。
- 用户选择跨会话持久化。

### 8.11 轻量设置中心

MVP 应提供统一设置中心，但只覆盖插件化工作台闭环需要的最小设置能力。

要求：

- 设置中心由宿主提供标准容器，负责打开、关闭、导航、焦点管理和错误边界。
- 设置内容来自已启用插件声明的 `settings-panel` contribution。
- MVP 至少包含插件、外观和搜索三个设置面板。
- 插件面板展示插件名称、ID、版本、启用状态、贡献能力和权限摘要，MVP 可先只读。
- 外观面板承载主题和背景选择，选择结果写入 workspace 并跨会话恢复。
- 搜索面板承载默认搜索源和搜索源启用状态，具体搜索源来自 `search-provider` contribution。
- 单个设置面板渲染失败时，只显示该面板错误状态，不影响设置中心其他面板。
- MVP 不做完整偏好设置产品，不做设置搜索、多 workspace、导入导出、插件安装升级和复杂权限审计。

### 8.12 弹窗和全屏视图

插件声明支持弹窗或全屏时，用户可以打开对应视图。

要求：

- 插件只声明 view ID。
- 平台打开标准宿主容器。
- 插件内容渲染在宿主内容区。
- 插件不能直接创建全局弹窗或全屏视图。

### 8.13 插件错误隔离

插件渲染失败时，只影响失败插件实例。

要求：

- Widget 渲染失败时显示错误卡片。
- Modal / fullscreen 渲染失败时显示局部错误状态。
- 同一工作台上的其他插件实例继续可用。
- 错误状态保留实例 ID，便于定位。

### 8.14 本地持久化

工作区、实例、配置和插件数据需要跨会话保存。

要求：

- 使用 IndexedDB 本地存储。
- 工作区装配状态、插件实例、插件数据分表存储。
- 插件业务数据按插件和实例命名空间隔离。
- 页面关闭并重新打开后恢复工作区、实例、尺寸、位置、插件配置和插件数据。

### 8.15 插件 UI 一致性

MVP 应提供 `@tabora/ui` 基础组件包，作为官方插件和未来第三方插件的 UI 基线。

要求：

- `@tabora/ui` 面向插件内容区组件，不接管宿主级容器。
- `@tabora/ui` 可以依赖 `@tabora/theme` 和 `solid-js`，不能依赖 `@tabora/platform-kernel`、`@tabora/storage` 或具体官方插件。
- 组件必须使用 theme token 和 CSS custom properties，不硬编码大面积颜色。
- 组件必须内建 hover、focus-visible、disabled、loading、error 等基础状态。
- 组件必须提供清晰的 label / aria label 组合方式，满足键盘可访问。
- MVP 组件建议包含 `Button`、`IconButton`、`Input`、`Textarea`、`Select`、`Checkbox`、`Switch`、`SegmentedControl`、`Tabs`、`Tooltip`、`Field`、`Badge`、`InlineError`、`Spinner`、`EmptyState`、`ListRow` 和 `CardSection`。
- `WidgetCard`、`Modal`、`FullscreenHost`、`SettingsHost`、`WorkbenchRail` 等宿主级结构不进入 `@tabora/ui`；这些容器由 shell / host 统一提供，插件只渲染其中内容。
- 官方插件新增 UI 时优先使用 `@tabora/ui`，确需自定义时应只在插件内部补充局部样式。

## 9. 关键用户流程

### 9.1 首次启动

1. 平台内核启动。
2. 发现官方插件。
3. 加载默认 workspace 模板。
4. 挂载默认主题、背景、工作台仪表盘布局、命令搜索和 widget 实例。
5. 用户看到包含左侧轻 rail、顶部命令搜索和主网格核心卡片的完整工作台。

成功标准：

- 页面可以直接使用。
- 默认内容来自官方插件。
- 首屏可见命令搜索、今日重点、快捷入口、便签和待办中的至少 3 个核心模块。
- 没有插件加载错误或空白页。

### 9.2 添加卡片

1. 用户打开添加卡片区域。
2. 系统列出已启用 widget contribution。
3. 用户选择 widget。
4. 平台创建实例并放入主网格。
5. Widget 插件渲染卡片视图。

成功标准：

- 新卡片立即出现。
- 多次添加同类 widget 会生成多个实例。
- 新实例跨会话保留。

### 9.3 调整卡片尺寸

1. 用户打开卡片尺寸选择器。
2. 系统展示该 widget 支持的尺寸。
3. 用户选择新尺寸。
4. 平台更新实例尺寸和网格跨度。
5. 卡片重新渲染。

成功标准：

- 不展示插件未声明支持的尺寸。
- 尺寸变化不会破坏其他卡片。
- 刷新后尺寸恢复。

### 9.4 卡片过多时继续使用工作台

1. 用户添加多个 widget 实例，主网格高度超过当前视口。
2. 新卡片追加到网格末尾。
3. 页面出现纵向滚动，用户可以向下访问低优先级卡片。
4. 用户把高频卡片拖到上方。
5. 用户把低频卡片调整为 S 或 M 尺寸。
6. 刷新页面后顺序、尺寸和位置恢复。

成功标准：

- 卡片不会被强行缩小到不可读。
- 不出现横向滚动。
- 添加新卡片不会打乱已有卡片顺序。
- 首屏核心卡片仍然优先可见。

### 9.5 使用命令搜索

1. 用户输入查询。
2. 命令搜索插件展示当前搜索源和基础快捷建议。
3. 用户提交查询。
4. 搜索插件根据当前搜索源生成目标 URL。
5. 搜索插件通过权限桥请求打开外部 URL。
6. 宿主执行已批准的打开动作。

成功标准：

- 空查询不会触发搜索。
- 外部打开通过权限桥发生。
- 搜索入口保持在首屏稳定位置。
- 插件不直接依赖宿主内部实现。

### 9.6 使用轻导航 Rail

1. 用户点击 rail 的添加卡片入口。
2. 宿主打开添加卡片面板。
3. 用户点击 rail 的插件入口。
4. 宿主打开插件设置面板或插件管理视图。
5. 用户点击 rail 的设置入口。
6. 宿主打开 settings host。

成功标准：

- Rail 入口稳定、可聚焦、可通过键盘访问。
- Rail 操作打开的是宿主容器，具体内容仍来自插件 contribution。
- 移动端折叠后不产生横向滚动。

### 9.7 插件失败恢复

1. 某个插件视图渲染失败。
2. 宿主边界捕获错误。
3. 失败实例显示错误回退。
4. 其他实例继续可用。

成功标准：

- 不出现整页白屏。
- 失败状态能定位到实例。
- 用户仍可操作其他卡片。

### 9.8 打开设置中心

1. 用户点击设置入口。
2. 宿主打开 settings host。
3. 宿主读取已启用插件贡献的 `settings-panel`。
4. 设置中心显示面板导航和默认面板。
5. 用户切换到外观、搜索或插件面板。
6. 宿主通过 view registry 渲染对应设置面板。

成功标准：

- 设置入口集中，不散落成多个互不关联的临时面板。
- 设置面板来自插件 contribution，而不是平台硬编码业务设置。
- 设置面板失败时，不影响工作台和其他设置面板。

## 10. 权限与安全需求

MVP 面向官方内置插件和本地可信插件，但权限模型必须从 V1 开始建立。

权限类型：

- `storage`：读写插件私有存储。
- `workspace`：读写工作区状态。
- `network`：访问外部网络。
- `clipboard`：读写剪贴板。
- `local-file`：通过用户授权流程读写本地文件。
- `external-open`：打开外部 URL。

授权原则：

- 默认允许插件私有存储、读取自身配置、注册 manifest 已声明的 contribution。
- 外部 URL 打开必须经过 `external-open` 权限。
- 未来 clipboard、本地文件、workspace write 等能力需要使用时授权。
- V2 前不接入不可信远程插件沙箱。

## 11. 数据与持久化需求

核心数据对象：

- `PluginRecord`：插件身份、版本、来源、启用状态、manifest 和授权权限。
- `Workspace`：当前布局、主题、背景和区域装配状态。
- `RegionState`：区域可接受扩展点和实例引用。
- `PluginInstance`：实例 ID、插件 ID、贡献 ID、区域、尺寸、网格位置和配置。
- `PluginDataRecord`：插件业务数据。

存储规则：

- 插件身份和权限不存页面位置。
- 插件实例存实例配置和位置，不存业务数据。
- 工作区存装配状态，不存具体业务内容。
- 插件业务数据按插件和实例隔离。

## 12. 验收标准

MVP 验收条件：

- 打开应用后看到完整默认工作台，且它由官方插件装配出来。
- MVP 提供至少两种布局插件（仪表盘式和流式），且它们有显著不同的区域结构。
- 用户可以在设置中切换布局，切换后工作台壳体正确重渲染。
- 切换布局后，所有插件实例数据保留。无法匹配区域的实例进入待放置状态。
- 当活跃布局插件渲染失败时，平台回退到最小安全布局，用户不面对白屏。
- 默认工作台下，首屏包含今日重点、快捷入口、便签和待办中的核心模块。
- 所有布局满足 2.1.3 节「全局可达性约束」——搜索、添加卡片、插件管理、设置在所有布局下可达。
- 平台代码不硬编码 Google URL、便签 UI、壁纸分类或待办行为等具体业务功能。
- 官方功能通过 manifest、contribution、runtime context、registry、permissions 和 plugin storage 实现。
- 官方插件内容区优先使用 `@tabora/ui` 基础组件，保持控件状态和可访问性一致。
- 用户可以添加同一个 widget 插件的多个实例。
- 用户可以从插件支持尺寸列表中选择每个 widget 实例的尺寸。
- 当卡片数量超过一屏时，主网格可以纵向滚动，且不会出现横向滚动或不可读压缩。
- 插件声明支持弹窗或全屏时，用户可以打开对应视图。
- 用户可以通过插件 contribution 切换主题和背景。
- 用户可以打开轻量设置中心，并在其中查看插件面板、切换主题背景、配置默认搜索源。
- 插件渲染错误只影响失败的插件实例。
- 关闭并重新打开后，工作区、实例、尺寸、位置、插件配置和插件数据完整恢复。

质量验收：

- `pnpm check` 通过。
- `pnpm test` 通过。
- `pnpm build` 通过。
- playground dev server 可以返回 HTTP 200。

## 13. 指标建议

MVP 阶段以功能闭环和稳定性为主，建议跟踪：

- 首屏成功渲染率。
- 默认插件激活成功率。
- 插件实例渲染失败次数。
- 工作区恢复成功率。
- 添加 widget 成功率。
- 搜索提交成功率。
- 外部打开权限拒绝次数。
- IndexedDB 读写失败次数。

后续生态阶段可增加：

- 插件安装量。
- 插件启用率。
- 插件崩溃率。
- 插件平均留存。
- 用户自定义工作区数量。

| 平台和官方插件边界不清     | 平台重新耦合业务功能   | 所有具体能力必须通过 manifest 和 contribution 进入                              |
| 插件权限模型后补困难       | 未来第三方生态成本高   | MVP 即建立 permissions bridge                                                   |
| 插件错误导致整页不可用     | 影响基础可用性         | 宿主渲染边界隔离失败实例                                                        |
| IndexedDB 数据结构演进困难 | 后续迁移风险高         | 数据按 workspace、instance、pluginData 分层                                     |
| 默认体验太空               | 用户无法理解产品价值   | 官方插件包提供完整默认工作台                                                    |
| MVP 过度像普通新标签页     | 产品差异化不足         | 默认布局使用轻 rail 和今日重点建立工作台心智                                    |
| 今日流能力过早膨胀         | MVP 范围失控           | 只做今日重点卡片，不做日程、跨插件聚合和复杂任务流                              |
| 插件 UI 风格分叉           | 官方和第三方体验不一致 | MVP 提供 `@tabora/ui`，插件优先复用基础控件                                     |
| 设置能力过早膨胀           | 分散 MVP 验证重点      | MVP 只做轻量 settings host 和必要面板，重功能后移                               |
| 过早做插件市场             | 分散 MVP 验证重点      | 市场、沙箱、审核延后到 V2                                                       |
| 布局插件过度自由           | 用户体验碎片化         | 2.1 节约束体系明确布局插件必须满足全局可达性、响应式安全、卡片底线和主题合规    |
| 布局切换时实例数据丢失     | 用户数据损失           | 切换前备份完整 workspace snapshot；旧区域无法匹配的实例保留在待放置列表而非丢弃 |
